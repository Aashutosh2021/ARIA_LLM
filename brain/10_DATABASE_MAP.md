# Database Map

The ARIA-LLM application has no standard relational database (like PostgreSQL/SQLite) or document store. Instead, it relies on file-based serialization structures stored in the workspace:

## File-based Data Stores

### 1. Training Corpora (`data/`)
- **`data/conversations.txt`**: Large synthetic conversational dataset containing prompt pairs generated by `scripts/prepare_conversations.py`. Used for finetuning local models.
- **`data/dialogs.txt`**: Cleaned, formatted dialogue turns dataset parsed from raw tab-separated inputs.
- **`data/tiny_corpus.txt`**: Small text dataset for sanity test runs.
- **`data/tinystories.txt`**: Pretraining text corpus.
- **`data/Conversation.csv`**: Conversational training dataset.
- **`data/intents.json`**: Intent conversational maps.

### 2. Checkpoints Registry (`checkpoints/`)
Saves training models serialized using `torch.save()`. Checkpoint states are structured as a dictionary containing:
- **`model_state`**: State dict of active weights (maps names to weights tensors).
- **`optimizer_state`**: AdamW optimizer values.
- **`scheduler_state`**: Warmup cosine scheduler steps.
- **`step`**: Global step count.
- **`epoch`**: Active epoch count.
- **`config`**: GPT configuration variables (vocab size, layer count, activation, etc.) used to dynamically reconstruct the model prior to weights loading.

### 3. Vocab and Tokenizer Stores (`checkpoints/vocab.json`)
The trained tokenizers serialize their states to a JSON registry:
- **`vocab`**: Maps string tokens to token IDs.
- **`special_tokens`**: Maps special identifiers (e.g. `<PAD>`, `<UNK>`, `<BOS>`, `<EOS>`, `<MASK>`) to their specific token IDs.
- **`merges`**: Ordered BPE byte pair merges.
